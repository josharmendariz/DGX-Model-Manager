Reference

A unified interface for managing AI models and inference engines on the NVIDIA DGX Spark. Pull Ollama models, browse HuggingFace, manage your local model inventory, route through LiteLLM, and control SGLang, vLLM, llama.cpp, LocalAI, and ComfyUI — all from one browser tab.

01 — Overview

How the Stack Fits Together

Every inference request on your device flows through one of several paths. LiteLLM sits in front of OpenAI-compatible backends and presents a single unified API at :4000 to all clients. Specialized engines like ComfyUI have their own web interface.

The Model Manager app (:8090) sits alongside this stack and gives you a GUI to control it — it never sits in the request path. It talks to each service's API and to Docker directly to manage container lifecycle.

Key principle

Engines like SGLang and vLLM are each dedicated to one large model at a time and take ~5 minutes to load. Ollama handles smaller models and can switch instantly. ComfyUI has its own web interface for image generation workflows. Use the right tool for the right job.

02 — Model Management

Ollama Models

The Ollama tab is your primary model store. Pull any model from the Ollama library by name, see what's installed with disk usage, and delete models you no longer need.

Pulling a Model

1
Type the model name
Enter the model name in the pull box — use the exact names from ollama.com/library. Include the tag for a specific size, e.g. qwen2.5:7b. Without a tag, Ollama pulls the default (usually the smallest).
2
Watch the progress bar
The bar shows live download progress with MB counts. Large models (30B+) can take several minutes depending on your connection. Don't close the tab.
3
Model appears in the grid
The installed models grid refreshes automatically when the pull completes. If LiteLLM wildcard routing is active, the model is immediately available to all apps at :4000.

Model Name Format

Model name patterns
# Name only — pulls default (smallest) tag llama3.2 phi4 gemma3 # Name:tag — pulls specific size qwen2.5:7b qwen2.5:32b qwen2.5-coder:32b deepseek-r1:14b # Quantization variant qwen2.5:32b-instruct-q4_K_M # Check available tags at: # https://ollama.com/library/<model-name>

Deleting a Model

Click the ✕ button on any model card. You'll be asked to confirm — deletion is permanent and frees the disk space immediately. The model list refreshes automatically.

VRAM headroom

A large model running via SGLang or vLLM can use ~97 GB or more of the 128 GB unified memory. That may leave only ~31 GB for Ollama models. Keep this in mind when pulling large models — stop the engine first if you want to run a 30B+ model via Ollama.

03 — Model Management

Model Inventory

The Inventory tab provides a unified view of every model on your device — HuggingFace cache models, custom directory models, and Ollama models — all in one searchable, filterable table.

Browsing Your Models

The table shows each model's name, task type, format, dtype, parameter count, disk size, source, and any engine profile that references it. Use the toolbar to narrow the view:

1
Search
Type any part of a model name or owner in the search box. Results filter in real-time as you type.
2
Filter by Source / Format / Task
Use the dropdowns to show only HF Cache, Custom Dir, or Ollama models. Filter by format (safetensors, GGUF, PyTorch) or task type (Text Gen, Vision LLM, Embedding, TTS, etc.).
3
Sort
Sort the table by Name, Size, or Parameter count using the sort dropdown.

HuggingFace Enrichment

Click Fetch HF Info to enrich all HF-sourced models with metadata from HuggingFace: pipeline tags, download counts, and likes. This data is cached locally for 7 days to avoid rate limiting.

Custom Directories

Expand the Custom Directories section to add any folder on your device to the inventory scan. Models in custom directories appear alongside HF cache models with a "Custom" source badge. Useful for models stored outside the default HF cache location.

Stats bar

The stats bar at the top of the table shows the total model count, combined disk usage, and a breakdown by source — giving you a quick overview of your model footprint.

04 — Model Management

HuggingFace Browse

Search and discover models on HuggingFace Hub without leaving the app. The HF Browse tab lets you find models, inspect their files, discover quantized variants, and download with one click.

Searching

Enter a query in the search box, optionally filter by pipeline type (Text Generation, Image Classification, etc.), and sort by downloads, likes, trending, or recent. Results appear as cards with task badges, download counts, and format indicators.

Exploring a Model

Click any result card to expand it. This loads the model's file list with individual file sizes, plus automatically searches for quantized variants — GGUF, GPTQ, and AWQ versions of the same model published by community quantizers.

Downloading

Click the Download button on any model or variant. This pre-fills the HF Download tab with the repository ID and switches to it, so you can start the download immediately.

Variant discovery

When you expand a model like meta-llama/Llama-3.3-70B-Instruct, the browser automatically searches for community GGUF and quantized versions. This helps you find smaller variants that fit in memory without hunting through HuggingFace manually.

05 — Model Management

HuggingFace Download

Downloads model weights directly from HuggingFace Hub to your device. Use this for models you want to run via SGLang, vLLM, or other engines — models that need to be loaded at server startup rather than pulled on demand.

Repository ID Format

HuggingFace repo ID format
# Format: owner/model-name mistralai/Mistral-Small-4-119B-2603-NVFP4 Qwen/Qwen3-Coder-Next meta-llama/Llama-3.3-70B-Instruct deepseek-ai/DeepSeek-V3

Where Files Land

Without a custom local directory, downloads go to ~/.cache/huggingface/hub/<owner>--<model>/ — the standard HF cache location. Engine startup scripts typically reference this path. If you specify a custom directory, update your startup script to point to that path.

Custom directory requirement

To download to a custom path, the directory must first be registered in the Inventory → Scan Directories section. The app validates download destinations against the HF cache and your registered directories — unregistered paths are rejected. Add the directory once, then use it for downloads freely.

Large model downloads

A 70B model is ~140 GB at BF16. NVFP4 models are much smaller (~70 GB for 119B). The progress log streams stdout from huggingface_hub.snapshot_download. If the connection drops mid-download, re-running will resume from where it left off — HF Hub supports partial downloads.

06 — Services

LiteLLM Routing

LiteLLM is the single API endpoint all your apps talk to. This tab shows you what models are currently routed, lets you apply the wildcard config, and gives you a live view of the config file.

Wildcard Routing

By default, every Ollama model you pull requires a manual entry in litellm_config.yaml before other apps can see it. The wildcard button fixes this permanently with one click.

What the wildcard button adds to your config
model_list: # Your existing SGLang entry stays unchanged - model_name: Mistral-Small-4 litellm_params: model: openai/Mistral-Small-4 api_base: http://127.0.0.1:30000/v1 # Wildcard — auto-exposes every Ollama model - model_name: ollama/* litellm_params: model: ollama/* api_base: http://127.0.0.1:11434 litellm_settings: drop_params: true

After applying, LiteLLM restarts automatically. Every model you pull into Ollama from that point on is instantly available — no more config edits, ever.

Active Routes

The routes grid shows every model currently exposed at :4000. This is what your apps see when they call GET /v1/models. Orange tags are SGLang-backed, green tags are Ollama-backed.

Restarting LiteLLM

The Restart button calls sudo systemctl restart litellm. Use this after any manual edits to the config file. LiteLLM takes about 3 seconds to come back up.

07 — Services

Inference Engines

The app manages five inference engines through a unified interface. Each engine has its own tab with a status card, profile selector, and start/stop controls. All engines use the same profile system — shell scripts auto-discovered from a directory.

Engine Overview

Engine Port Use Case API
SGLang :30000 High-performance LLM inference. RadixAttention, fast structured output, strong MoE support. OpenAI-compatible
vLLM :8000 Production LLM inference. Broad model/quantization support, PagedAttention. OpenAI-compatible
llama.cpp :8080 GGUF model inference. Lightweight, ARM-optimized, runs without Docker. OpenAI-compatible
LocalAI :9090 Multi-modal AI — LLM, TTS, STT, and image generation in one engine. OpenAI-compatible
ComfyUI :8188 Image generation workflows with a node-based web UI. Own web UI

Status Card

Each engine tab shows a status card with a colored LED: green when running, red when stopped. For OpenAI-compatible engines, the currently loaded model name is pulled live from the engine's /v1/models endpoint. ComfyUI shows a running/stopped status and an Open UI button that links to its web interface.

Stop before starting

Always stop the running container before starting a new profile. Starting a new profile while one is already running will fail — the port will be in use.

Starting an Engine

Select a profile from the dropdown and click Start. The start script runs in the background — the app polls the engine's health endpoint every 20 seconds and updates the status card when it comes online. The ready toast only fires once the engine is actually serving, not just when the container starts. If the engine doesn't come up within 10 minutes, the poll times out automatically.

SGLang vs vLLM

Both are excellent choices for large model inference. SGLang has RadixAttention (prefix caching) and faster structured output. vLLM has broader quantization support and a mature ecosystem. Both expose the same OpenAI-compatible API (/v1/), so your apps don't need to change when switching between them.

Port conflicts

Each engine uses a different default port, so they can all run simultaneously — provided you have enough memory. SGLang at :30000, vLLM at :8000, llama.cpp at :8080, LocalAI at :9090, ComfyUI at :8188.

08 — Services

Engine Profiles

All five engines use the same profile system. The app auto-discovers shell scripts named start_*.sh from each engine's directory. No JSON editing required — adding a new profile is just adding a new script.

Engine Script Directory
SGLang ~/SGLang/start_*.sh
vLLM ~/vLLM/start_*.sh
llama.cpp ~/llama.cpp/start_*.sh
LocalAI ~/LocalAI/start_*.sh
ComfyUI ~/ComfyUI/start_*.sh

Creating a Profile

1
Create a startup script in the engine directory
Create a script named start_*.sh in the engine's directory (e.g. ~/SGLang/start_qwen3_70b.sh). The filename must start with start_ and end with .sh.
2
Add header comments (optional but recommended)
Add # Name:, # Description:, and # VRAM: comments in the first 20 lines. These control what the profile card displays in the UI. Without them, the name is derived from the filename.
3
Open the engine tab
The new profile appears in the selector immediately. No app restart, no config editing. The UI re-scans on every tab load.

Example SGLang Script

~/SGLang/start_mistral_small4.sh
#!/usr/bin/env bash # Name: Mistral Small 4 # Description: 119B NVFP4 · ~5 min warm-up # VRAM: 119 sudo docker run --rm --gpus all --ipc=host \ --name my-sglang-container \ -v ~/.cache/huggingface:/root/.cache/huggingface \ -e TRITON_PTXAS_PATH=/usr/local/cuda/bin/ptxas \ -p 30000:30000 \ lmsysorg/sglang:nightly-dev-cu13 \ python3 -m sglang.launch_server \ --model-path /root/.cache/huggingface/hub/models--mistralai--Mistral-Small-3.2-24B-Instruct-2506/snapshots/main \ --quantization modelopt_fp4 \ --host 0.0.0.0 \ --port 30000 \ --tool-call-parser mistral

Example vLLM Script

~/vLLM/start_nemotron3super_vllm.sh
#!/usr/bin/env bash # Name: Nemotron 3 Super # Description: 49B BF16 · Nvidia flagship · ~5 min warm-up # VRAM: 97 sudo docker run --rm --gpus all --ipc=host \ --name my-vllm-container \ -v ~/.cache/huggingface:/root/.cache/huggingface \ -p 8000:8000 \ vllm/vllm-openai:latest \ --model /root/.cache/huggingface/hub/models--nvidia--Nemotron-Super-49B-v1/snapshots/main \ --host 0.0.0.0 \ --port 8000 \ --tensor-parallel-size 1 \ --tool-call-parser mistral \ --enable-auto-tool-choice

LiteLLM integration

To route engine models through LiteLLM, add an entry to litellm_config.yaml with model: openai/your-model-name and api_base pointing to the engine's URL. The Ollama wildcard only covers Ollama — each engine model needs an explicit entry.

09 — System

Settings

The Settings tab provides centralized configuration for the app and all connected services.

Service URLs

Configure the URL for each service. All seven services (Ollama, LiteLLM, SGLang, vLLM, llama.cpp, LocalAI, ComfyUI) have URL inputs with individual Test buttons for connectivity checks. Use Test All to verify all services at once. Changes are saved to config.json.

API Key

Set or clear the optional API key for authentication. See the Security section for details.

Runtime changes

All settings take effect immediately — no app restart required. Changes are persisted to config.json for next startup.

10 — System

Security

API key authentication is optional and can be enabled from the Settings tab. When a key is set, all mutating operations require the key.

What's Protected

Requires key Start/stop engines, pull/delete Ollama models, HF download, update config, apply wildcard, restart LiteLLM, clear logs, view app/engine/LiteLLM logs, view debug config, view system overview, view Docker containers, view LiteLLM config
No key needed Status checks, model inventory, profiles, HF search, node info

First-Boot Warning

If the app starts with no API key set and is binding to a non-loopback address (anything other than 127.0.0.1), it emits a loud warning to the systemd journal and waits 10 seconds before accepting connections. This gives you time to notice the open-access condition before the app is reachable. Set MODEL_MANAGER_ALLOW_UNAUTH=1 in the service environment to suppress the warning and delay if you intentionally want open access.

How It Works

The API key is sent as an Authorization: Bearer <key> header with every request that requires authentication. The app stores only the SHA-256 hash of the key in config.json — the plaintext key is never written to disk. Verification uses constant-time hmac.compare_digest() on the hashes to prevent timing attacks.

Remember your key

Since the key is stored as a hash, it cannot be recovered. If you lose the key, clear it by editing config.json and removing the api_key field, then restarting the app.

11 — System

Logs & Debug

The Logs & Debug tab provides full-stack visibility into your AI infrastructure without SSH access.

System Overview

A grid of stat cards showing hostname, IP address, architecture, total memory, Python version, app uptime, and disk usage for the HF cache directory. Health checks for all services show colored status (green/red/amber) with response latency in milliseconds. Sudo and Docker permission status are also displayed.

Running Configuration

Collapsible sections showing the current app configuration, parsed LiteLLM YAML, and discovered profiles for all engines. Useful for verifying what the app sees without checking files on the server.

Application Logs

A color-coded log viewer connected to the app's in-memory ring buffer (500 entries). Filter by severity level (DEBUG, INFO, WARNING, ERROR), search by text, and toggle 3-second auto-refresh. The buffer captures app events like engine starts, config changes, auth failures, and download activity. Logs are in-memory only — they clear on app restart.

Engine Logs

Tabbed viewer showing the latest log file for each engine from /tmp/. Each engine writes its stdout/stderr to a timestamped log file when started via the app. Switch between engines using the tab bar.

LiteLLM Logs

Reads the LiteLLM service journal via journalctl -u litellm. Shows the most recent log lines with search and auto-refresh. If the app doesn't have journal access, a permission error is displayed gracefully.

Docker Containers

A table showing all running Docker containers with ID, name, image, status, ports, and creation time. Click Refresh to update the view.

Smart scroll

Log panes only auto-scroll to the bottom if you haven't scrolled up. This lets you read through logs without being interrupted by new entries during auto-refresh.

MoE models and VRAM

MoE models like qwen3-coder-next (80B total, 3B active) load all expert weights into memory but only compute a fraction per token. The full 80B BF16 model needs ~160 GB — more than your device has. Use the quantized Ollama version which lands around 45 GB, comfortably fitting with SGLang stopped.

13 — Reference

Troubleshooting

Service Status Pills Show Red

Any engine red The engine container is stopped. Use its tab to start a profile, or check docker ps via SSH.
Ollama red Ollama service is down. Run sudo systemctl restart ollama via SSH.
LiteLLM red LiteLLM service is down. Use the Restart button on the LiteLLM tab or sudo systemctl restart litellm.

Pull Gets Stuck / No Progress

Bar stays at 0% Ollama may be downloading a manifest before the first layer. Wait 30s — it should start.
Connection error Check that Ollama is running. Your device needs internet access for pulls — verify via curl https://ollama.com.
Model not found Double-check the exact name at ollama.com/library. Names are case-sensitive.

HF Download Fails with Permission Error

403 on custom path The destination directory must be registered under Inventory → Scan Directories before it can be used as a download target. Add the directory there first, then retry the download.
No error but nothing downloads Check the browser console for a failed fetch. If the API key is set, ensure it's stored in your browser — open Settings and re-enter it if needed.

Models Not Showing in LiteLLM Routes

Wildcard not applied Go to the LiteLLM tab and click Apply Wildcard. This is a one-time step.
LiteLLM needs restart Even with the wildcard, LiteLLM needs to be running. Check the status pill and restart if red.
Model just pulled With the wildcard active, newly pulled models are available immediately — no restart needed.

Engine Fails to Start

Port already in use A previous container is still running on the same port. Use the Stop button first, wait 10s, then start again.
Container name conflict If a stopped container with the same name exists, add --rm to your docker run or run docker rm <name> first.
Model path missing Check that the model was fully downloaded. Run ls ~/.cache/huggingface/hub/ via SSH.
OOM error Not enough unified memory. Stop other engines first, or reduce memory flags in the start script (e.g. --gpu-memory-utilization for vLLM, --mem-fraction-static for SGLang).
No profiles found Make sure start scripts are in the correct directory (e.g. ~/SGLang/start_*.sh) and start with start_.
Start timeout The app polls for 10 minutes. If the engine doesn't become healthy, check the engine log in the Logs & Debug tab for errors.

Mistral Returns HTTP 400 / "Message Ordering Conflict"

Consecutive same-role messages Mistral enforces strict user/assistant role alternation. Any middleware, agent framework, or callback that injects consecutive messages with the same role (e.g. two user turns in a row) will trigger an HTTP 400. Audit your LiteLLM callbacks and any agent boot hooks.
Subagent tool-call injection Subagent tool-call/result sequences may violate the alternation rule if injected raw into the conversation. Ensure tool results are wrapped in the expected alternating structure before forwarding to Mistral.
Session compaction Context compaction that merges or reorders messages can also break alternation. If the error fires mid-conversation, check your compaction logic.

App Not Loading

Check service sudo systemctl status model-manager
View logs sudo journalctl -u model-manager -f
Restart sudo systemctl restart model-manager

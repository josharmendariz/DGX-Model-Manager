A unified interface for pulling, routing, and launching models across Ollama, LiteLLM, and SGLang on the HP ZGX Nano G1n.

01 — Overview

How the Stack Fits Together

Every inference request on the ZGX flows through one of three paths — Ollama for small/medium models, and SGLang or vLLM for the flagship large model. LiteLLM sits in front of all backends and presents a single OpenAI-compatible API at :4000 to all clients.

Clients

Open WebUI :8081

▼ all send to :4000/v1 ▼

↙ ↓ ↘
SGLang · :30000
Large models (engine A)
Ollama · :11434
All other models
vLLM · :8000
Large models (engine B)

The Model Manager app (:8090) sits alongside this stack and gives you a GUI to control it — it never sits in the request path. It talks to Ollama, LiteLLM, and Docker directly to manage state.

Key principle

SGLang and vLLM are each dedicated to one large model at a time and take ~5 minutes to load. Ollama handles everything else and can switch models instantly. Use the right tool for the right model size.

02 — Tab Guide

Ollama Models

The Ollama tab is your primary model store. Pull any model from the Ollama library by name, see what's installed with disk usage, and delete models you no longer need.

Pulling a Model

1
Type the model name
Enter the model name in the pull box — use the exact names from ollama.com/library. Include the tag for a specific size, e.g. qwen2.5:7b. Without a tag, Ollama pulls the default (usually the smallest).
2
Watch the progress bar
The bar shows live download progress with MB counts. Large models (30B+) can take several minutes depending on your connection. Don't close the tab.
3
Model appears in the grid
The installed models grid refreshes automatically when the pull completes. If LiteLLM wildcard routing is active, the model is immediately available to all apps at :4000.

Model Name Format

Model name patterns
# Name only — pulls default (smallest) tag llama3.2 phi4 gemma3 # Name:tag — pulls specific size qwen2.5:7b qwen2.5:32b qwen2.5-coder:32b deepseek-r1:14b # Quantization variant qwen2.5:32b-instruct-q4_K_M # Check available tags at: # https://ollama.com/library/<model-name>

Deleting a Model

Click the ✕ button on any model card. You'll be asked to confirm — deletion is permanent and frees the disk space immediately. The model list refreshes automatically.

VRAM headroom

A large model running via SGLang or vLLM can use ~97 GB or more of the 128 GB unified memory. That may leave only ~31 GB for Ollama models. Keep this in mind when pulling large models — stop the engine first if you want to run a 30B+ model via Ollama.

03 — Tab Guide

LiteLLM Routing

LiteLLM is the single API endpoint all your apps talk to. This tab shows you what models are currently routed, lets you apply the wildcard config, and gives you a live view of the config file.

Wildcard Routing

By default, every Ollama model you pull requires a manual entry in litellm_config.yaml before other apps can see it. The wildcard button fixes this permanently with one click.

What the wildcard button adds to your config
model_list: # Your existing SGLang entry stays unchanged - model_name: Mistral-Small-4 litellm_params: model: openai/Mistral-Small-4 api_base: http://127.0.0.1:30000/v1 api_key: sk-sglang # Wildcard — auto-exposes every Ollama model - model_name: ollama/* litellm_params: model: ollama/* api_base: http://127.0.0.1:11434 litellm_settings: drop_params: true

After applying, LiteLLM restarts automatically. Every model you pull into Ollama from that point on is instantly available — no more config edits, ever.

Active Routes

The routes grid shows every model currently exposed at :4000. This is what your apps see when they call GET /v1/models. Orange tags are SGLang-backed, green tags are Ollama-backed.

Restarting LiteLLM

The Restart button calls sudo systemctl restart litellm. Use this after any manual edits to the config file. LiteLLM takes about 3 seconds to come back up.

04 — Tab Guide

SGLang Engine

SGLang is the high-performance inference engine for large models. It loads one model at a time into the GB10 unified memory pool. Unlike Ollama, model swaps require a full restart (~5 minutes).

Status Card

The green LED and top bar glow indicate SGLang is running and responsive. The model name shown is pulled live from :30000/v1/models. The Stop button sends docker stop sglang.

Stop before starting

Always stop the running container before starting a new profile. Starting a new profile while one is already running will fail — the port will be in use.

Profiles

Profiles are pre-defined startup configurations stored in ~/model-manager/sglang_profiles.json. Each profile points to a shell script that starts SGLang with the right flags for that model.

~/model-manager/sglang_profiles.json
[ { "id": "mistral-small-4", "name": "Mistral Small 4 · 119B NVFP4", "script": "~/SGLang/start_sglang.sh", "description": "Primary flagship · ~97 GB VRAM · ~5 min warm-up", "vram_gb": 97, "model_path": "~/.cache/huggingface/hub/..." } ]

To add a new profile for a different large model, duplicate the block, give it a unique id, and point script at a new startup script. The profile will appear in the UI immediately on next page load.

Starting a Profile

Select a profile and click Start. The start script runs in the background — the app polls SGLang's /health endpoint every 20 seconds and updates the status card when it comes online. Expect 3–7 minutes for large models.

05 — Tab Guide

vLLM Engine

vLLM is a high-performance inference engine and an alternative to SGLang. Like SGLang, it loads one model at a time and runs as a Docker container. The vLLM tab works identically to the SGLang tab — same profile system, same status polling, same start/stop workflow.

Status Card

The green LED and top bar glow indicate vLLM is running and responsive. The model name is pulled live from :8000/v1/models. The Stop button sends docker stop vllm.

Container naming

Your vLLM start script must name the container vllm (i.e. --name vllm) for the app to detect and control it. This mirrors the SGLang convention where the container must be named sglang.

Profiles

Profiles are stored in ~/model-manager/vllm_profiles.json. Each profile points to a shell script that starts vLLM with the right flags for that model. See the vLLM Profiles reference section for the full format and example scripts.

Starting a Profile

Select a profile and click Start. The start script runs in the background — the app polls vLLM's /health endpoint every 20 seconds and updates the status card when it comes online. The ready toast only fires once the model is actually serving, not just when the container starts.

SGLang vs vLLM

Both engines expose the same OpenAI-compatible API (/v1/), so your apps don't need to change when switching between them. Use whichever fits your workflow — or run both on different ports with different models if you have the memory.

Default ports

SGLang defaults to :30000, vLLM defaults to :8000. These don't conflict, so both can run simultaneously if you have enough memory for two large models.

06 — Tab Guide

HuggingFace Download

Downloads model weights directly from HuggingFace Hub to the ZGX. Use this for models you want to run via SGLang or vLLM — models that need to be loaded at server startup rather than pulled on demand.

Repository ID Format

HuggingFace repo ID format
# Format: owner/model-name mistralai/Mistral-Small-4-119B-2603-NVFP4 Qwen/Qwen3-Coder-Next meta-llama/Llama-3.3-70B-Instruct deepseek-ai/DeepSeek-V3

Where Files Land

Without a custom local directory, downloads go to ~/.cache/huggingface/hub/<owner>--<model>/ — the standard HF cache location. SGLang expects models here by default. If you specify a custom directory, remember to update your SGLang startup script to point to that path.

Large model downloads

A 70B model is ~140 GB at BF16. NVFP4 models are much smaller (~70 GB for 119B). The progress log streams stdout from huggingface_hub.snapshot_download. If the connection drops mid-download, re-running will resume from where it left off — HF Hub supports partial downloads.

MoE models and VRAM

MoE models like qwen3-coder-next (80B total, 3B active) load all expert weights into memory but only compute a fraction per token. The full 80B BF16 model needs ~160 GB — more than the ZGX has. Use the quantized Ollama version which lands around 45 GB, comfortably fitting with SGLang stopped.

08 — Reference

Adding SGLang Profiles

When you download a new large model via the HF tab and want to serve it through SGLang, the process is:

1
Create a startup script
Copy ~/SGLang/start_sglang.sh, rename it for the new model, and change the model path and any flags (e.g. --tool-call-parser may differ per model family).
2
Add a profile block to sglang_profiles.json
Edit ~/model-manager/sglang_profiles.json and add a new object with a unique id, name, script path, description, and vram_gb.
3
Reload the SGLang tab
The new profile appears in the profile selector immediately. No app restart needed.

SGLang Flags by Model Family

Model FamilyRequired FlagsNotes
Mistral (all) --tool-call-parser mistral
--moe-runner-backend flashinfer_cutlass		 MoE runner only needed for MoE variants. Always use --attention-backend triton on GB10.
Qwen3-Coder-Next --tool-call-parser qwen3_coder New parser — different from qwen2.5. Requires SGLang ≥ v0.5.8.
All models on GB10 --attention-backend triton FlashInfer has GB10 bugs. Never use --quantization modelopt_fp4 or --fp4-gemm-backend on SM121A.
09 — Reference

Adding vLLM Profiles

vLLM profiles work identically to SGLang profiles. When you download a new large model via the HF tab and want to serve it through vLLM:

1
Create a startup script
Write a shell script that runs docker run --name vllm ... with the correct model path and flags. The container must be named vllm.
2
Add a profile block to vllm_profiles.json
Edit ~/model-manager/vllm_profiles.json and add a new object with a unique id, name, script path, description, and vram_gb.
3
Reload the vLLM tab
The new profile appears in the profile selector immediately. No app restart needed.

Example vLLM Profile

~/model-manager/vllm_profiles.json
[ { "id": "llama3-70b", "name": "Llama 3.3 70B Instruct", "script": "~/vllm/start_llama3_70b.sh", "description": "Meta flagship · BF16 · ~140 GB · ~7 min warm-up", "vram_gb": 140, "model_path": "~/.cache/huggingface/hub/..." } ]

Example vLLM Start Script

~/vllm/start_llama3_70b.sh
#!/usr/bin/env bash sudo docker run --rm --gpus all --ipc=host \ --name vllm \ -v ~/.cache/huggingface:/root/.cache/huggingface \ -p 8000:8000 \ vllm/vllm-openai:latest \ --model /root/.cache/huggingface/hub/models--meta-llama--Llama-3.3-70B-Instruct/snapshots/main \ --host 0.0.0.0 \ --port 8000 \ --tensor-parallel-size 1

Common vLLM Flags

FlagDescription
--model Path to model weights or HuggingFace repo ID
--tensor-parallel-size Number of GPUs for tensor parallelism (1 for DGX Spark)
--max-model-len Maximum context length. Reduce if hitting OOM errors.
--gpu-memory-utilization Fraction of GPU memory to use (default 0.9). Lower to leave room for Ollama.
--dtype Data type: auto, float16, bfloat16
--quantization Quantization method: awq, gptq, squeezellm, etc.
--tool-call-parser Tool calling format: mistral, hermes, llama3_json, etc.
--enable-auto-tool-choice Auto-detect tool calls from model output
LiteLLM integration

To route vLLM models through LiteLLM, add an entry to litellm_config.yaml with model: openai/your-model-name and api_base: http://127.0.0.1:8000/v1. The Ollama wildcard doesn't cover vLLM — you need an explicit entry per model.

Troubleshooting

Service Status Pills Show Red

SGLang redSGLang container is stopped. Use the SGLang tab to start it, or check docker ps via SSH.
vLLM redvLLM container is stopped. Use the vLLM tab to start it, or check docker ps via SSH.
Ollama redOllama service is down. Run sudo systemctl restart ollama via SSH.
LiteLLM redLiteLLM service is down. Use the Restart button on the LiteLLM tab or sudo systemctl restart litellm.

Pull Gets Stuck / No Progress

Bar stays at 0%Ollama may be downloading a manifest before the first layer. Wait 30s — it should start.
Connection errorCheck that Ollama is running. The ZGX needs internet access for pulls — verify via curl https://ollama.com.
Model not foundDouble-check the exact name at ollama.com/library. Names are case-sensitive.

Models Not Showing in LiteLLM Routes

Wildcard not appliedGo to the LiteLLM tab and click Apply Wildcard. This is a one-time step.
LiteLLM needs restartEven with the wildcard, LiteLLM needs to be running. Check the status pill and restart if red.
Model just pulledWith the wildcard active, newly pulled models are available immediately — no restart needed.

SGLang Fails to Start

Port already in useA previous container is still running. Use the Stop button first, wait 10s, then start again.
Model path missingCheck that the model was fully downloaded. Run ls ~/.cache/huggingface/hub/ via SSH.
OOM errorNot enough unified memory. Stop other memory-heavy processes or reduce --mem-fraction-static in the start script.

vLLM Fails to Start

Port already in useA previous container is still running. Use the Stop button first, wait 10s, then start again.
Container name conflictMake sure your start script uses --name vllm. If a stopped container with that name exists, add --rm to your docker run or run docker rm vllm first.
Model path missingCheck that the model was fully downloaded. Run ls ~/.cache/huggingface/hub/ via SSH.
OOM errorNot enough unified memory. Stop SGLang if it's running, or reduce --gpu-memory-utilization in the start script.
Script not foundProfile script paths support ~/ but must point to an existing file. Verify with ls ~/vllm/start_*.sh.

App Not Loading

Check servicesudo systemctl status model-manager
View logssudo journalctl -u model-manager -f
Restartsudo systemctl restart model-manager
Quick health check

All services can be verified in one command: curl -s http://127.0.0.1:30000/health; curl -s http://127.0.0.1:8000/health; curl -s http://127.0.0.1:11434/api/tags >/dev/null && curl -s http://127.0.0.1:4000/health && echo "all ok"
